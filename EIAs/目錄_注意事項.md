目錄製作注意事項

1) 路徑與正規化
- 先將欲索引的檔案路徑 canonicalize（realpath / Resolve-Path），避免相對路徑或 symlink 導致跳脫目錄的安全風險。
- 在 Windows 上對比時注意大小寫與反斜線/正斜線差異（建議轉成統一的 "/" 並以 lower-case 比對）。

2) 權限檢查
- 所有讀檔動作應經 ACL wrapper（workspace/scripts/read_with_acl.js）驗證使用者是否有權限讀取該路徑。
- 不要在未檢查前直接把檔案內容暴露到公共目錄或訊息中。

3) 檔案展平 (flattening)
- 若某個章節資料夾只有一個子資料夾（例如 C07/auto/C07.md），目錄生成程序應該自動平展，避免多餘的一層路徑顯示。
- 若子資料夾係有意的分類（多個檔案），則保留結構。

4) 標題與摘要擷取
- 對 Markdown 檔案，先擷取第一個頂級標題（第一個以 `# ` 開頭的行）作為檔案 title。
- 如果沒有標題，降級為第一個非空行（並截斷到合理長度，例如 120 字元）。
- 若要附上簡短摘要，可擷取前 N 個非空行（建議 1–2 行）並去除長表格/HTML 標籤。

5) 編碼與大型檔案處理
- 讀取時指定 UTF-8；若失敗再嘗試常見編碼（Big5 / CP950），但要小心誤判。
- 避免一次把很大的檔案（>5MB）整段載入與嵌入目錄；對大型檔案只擷取前面 1–2KB 作標題/摘要。

6) 排序與呈現
- 章節按自然排序（C01、C02 ...），章內檔案按檔名或文件內序號排序。
- 目錄中顯示 "檔名 — 標題"，並提供檔案相對路徑以便直接開啟。

7) 排除與保密
- 明確列出要排除的檔案/資料夾模式（例如含敏感字、私密草稿、或暫存資料），避免意外曝光。
- 目錄生成腳本應支援一個排除清單（eg. .indexignore 或 config 參數）。

8) 日誌與稽核
- 每次生成或被要求讀檔時，將操作記錄到 access log（workspace/.openclaw/access.log），包含 user id、時間、目標路徑與結果（ok/denied/error）。

9) 可重現與更新時機
- 建議把生成動作做成可重複執行的腳本（scripts/generate_index.js），並在：
  a) 手動要求時執行、b) 上游檔案更新後（提交/同步後）執行、或 c) 排程（每日/每次部署後）執行。

10) 路徑顯示格式
- 目錄檔內路徑採相對於 outputs 的相對路徑（例如 C07/auto/C07.md），或統一為 POSIX 風格斜線以利跨平台閱讀。

11) 二進位檔與非 md 檔處理
- 若遇到 PDF、圖檔或其他二進位檔，目錄中僅列出檔名與檔案大小及建立日期，不嘗試嵌入內容。

12) 錯誤與回退
- 生成過程應保留上一次的目錄備份（目錄.md.bak），若本次失敗可回退。

13) 使用者互動
- 目錄檔頭部應標示生成時間與生成者（script 名稱與版本），並說明如何重新產生（例如：node scripts/generate_index.js）。

14) 自動平整與檢查 C06 特例
- 若某章節被發現為特殊結構（例如 C06 缺檔），在目錄中清楚標示為「缺檔／待轉檔」，不要自動忽略。

15) 安全備註
- 切勿在公開群組貼上完整敏感內容；若要分享整份文件，先確認對方有 read_with_acl 的讀取權限。

範例：如何重新產生目錄
- 在 workspace 根目錄執行：
  node scripts/generate_index.js
- 權限保證：所有遠端查詢應先由 read_with_acl.js 驗證：
  node scripts/read_with_acl.js --uid <sender_uid> --target <path>
